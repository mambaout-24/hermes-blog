嗨，我是 Lena。我的第一個 Hermes Cron Job 大概跑了三天，我才發現它根本沒在做我以為它在做的事。

它在清單裡看起來一切正常。排程是對的，提示詞也是某個星期二下午寫的，看著挺清楚。但執行的結果總感覺有點歪——不是說壞掉了，就是……跟我在平常對話裡問同一個問題得到的回答不太一樣。

我停下來想了想。我一直在把它當成 Linux 的 cron 來用。問題是，**Hermes Agent 的 cron 根本不是 Linux cron**。它只是借用了那個排程語法而已。

這篇文章記錄了我慢慢搞懂 Hermes 裡 cron 到底是怎麼運作的過程——以及為什麼那麼多任務看起來排程正確，執行起來卻不對勁。這種事情大概不急的時候都不覺得是問題，等覺得是問題的時候就來不及了。

---

## Hermes Cron 到底在做什麼

第一件值得大聲說清楚的事：**Hermes 的 cron job 不是一個計時器上的 shell 指令。它是一個按照排程觸發的 Agent 對話。**

這個區別比我想像的重要得多。根據官方文檔，每次排程執行都是在一個**全新的 Agent 對話**裡開始的。沒有任何對話歷史。沒有上一次執行的記憶。沒有從你平常互動中帶過來的上下文。Agent 執行工作需要的所有東西，都得靠提示詞提供——或者靠附加的技能（Skills）來提供。

第二件事：**Gateway 必須在運行**。排程器不是一個可以忽略的獨立守護行程。在 Gateway 模式下，排程器的 tick 是 Gateway 主事件循環的一部分，大概每 60 秒檢查一次。如果 Gateway 沒有啟動，你的任務就靜靜躺在 `jobs.json` 裡，看起來排程完美，但什麼都不做。這個我是意外發現的——一個看起來「沒有在跑」的任務其實沒問題，只是 Gateway 兩天前掛掉了。

第三件事是關於投遞（Delivery）的。Agent 的最終回應會自動投遞到配置好的目標。所以如果你在提示詞結尾寫了「把這個發到 Telegram」，而目標**確實**是 Telegram，Hermes 會檢測到重複然後只發一次。這部分不用你操心——但你得知道它在做這件事。

## 排程格式、重複、技能與投遞目標

Hermes 同時支援自然語言間隔和標準的 5 欄位 POSIX cron 表達式。所以 `every 1h`、`30m` 以及 `0 9 * * 1-5` 都可以用。

5 欄位格式跟其他地方用的是一樣的——Linux crontab、Kubernetes CronJobs、GitHub Actions——而 Wikipedia 上關於 cron 的條目其實是個出乎意料好用的參考資料，如果你想確認 Vixie 方言（Hermes 也在用的那個）裡步進值和範圍到底是怎麼互動的話。

如果你不確定自己的表達式到底是不是你想的那樣，`crontab.guru` 是我一直回頭用的工具。把表達式貼進去，讀它的中文解說，檢查接下來 5 次執行時間。這種事花 30 秒就能省下一個除錯的下午。

你可以附加技能（`--skill name`，一個或多個），用 `--deliver` 設定投遞目標，給任務取個名字，還可以按任務覆蓋模型。任務記錄以 JSON 格式儲存在 `~/.hermes/cron/jobs.json` 裡，使用原子寫入——意思是如果寫入被中斷，你也不會拿到一個半壞的檔案。

## 創建經得起實際使用的任務

我最後常用的模式大概是這樣的：

```bash
hermes cron create "0 9 * * 1-5" \
  "從 ~/work/project 拉取昨天的提交，\
   用 5 個要點總結變更內容，並標出任何看起來異常的地方" \
  --skill git-summary \
  --name "工作日站會準備"
```

這裡我想特別說三個點：

**1. 排程是明確的。** `0 9 * * 1-5`——早上 9 點，工作日。不是每天早上 9 點。這個 cron 表達式對任何懂語法的人來說都不會誤解，包括半夜 11 點試圖 debug 為什麼星期六觸發了的我自己。

**2. 提示詞是自包含的。** 它說了 repo 在哪。它說了要總結什麼。它說了要幾個要點。它沒有假設 Agent 記得之前的任何東西。

**3. 附加了技能。** `git-summary` 技能（這裡是假設的）承載了具體的操作步驟——怎麼讀 git log、用什麼格式、什麼算「異常」。Cron 的提示詞只需要說**做什麼**，不用說**怎麼做**。

第三點是我一直在反覆學習的東西。

## 為什麼自包含的提示詞比你想像的重要

這裡有一個我第一週寫的提示詞，它沒有按照我預期的方式工作：

> 「跟上次一樣，用這週的數據。」

在我平常的互動對話裡，這個提示詞完全沒問題。Agent 有對話歷史，知道「上次」是什麼，知道我們在看什麼數據。但在 cron 執行中，這些都不存在。全新的對話。沒有記憶。沒有上下文。Agent 讀到「跟上次一樣」，完全沒有東西可以比較。

一個真正能用的版本是這樣的：

> 「讀取 ~/data/weekly-signups.csv。計算相較於前 7 天的註冊人數變化百分比。報告數字、變化量以及前 3 大來源管道。格式化為 Slack 訊息。」

無聊。囉嗦。具體。每次都管用。

官方文檔裡有一個很好的例子，推薦的模式本質上是：**假設 Agent 是第一次讀你的提示詞，在隔離環境中，除了附加的技能之外沒有任何上下文。** 如果這個提示詞對一個陌生人來說不能作為一個獨立的指令，那麼它對一個全新的 cron 對話也同樣不合適。

## 常見的 Cron 失敗模式與預防方法

以下是一些我看過出問題的模式，大部分是我自己踩的坑：

**Gateway 沒在運行的沉默。** 任務已排程、清單有顯示、狀態看起來正確，但什麼都沒觸發。解決方法通常是：用 `hermes cron status` 確認排程器還活著，然後確保 Gateway 以服務形式安裝，這樣重新開機或登出後還能繼續跑。在你把它安裝成使用者或系統服務之前，每一個 `hermes gateway` 前台執行都離失敗只差一個終端機視窗關閉。

**帶有隱含上下文的脆弱提示詞。** 這個已經說過了。我最常看到的版本是「總結一下有什麼新的」——跟什麼比？什麼時候？Agent 不知道。

**因頻率誤判導致任務重疊。** 一個設定為每 1 分鐘執行一次但需要 90 秒才能完成的任務，會產生一個有趣的問題。Hermes 在排程器的 tick 上使用檔案鎖定，防止同一批到期的任務被並行處理兩次，這很好。但更深層的問題是：你**真的**需要 1 分鐘的精確度嗎？通常不需要。

**提示詞要求投遞導致的重複發送。** 如果你的提示詞結尾寫了 `send_message(...)` 到排程器已經要投遞的同一個目標，Hermes 會檢測到重複然後跳過第二次發送。這沒問題。但如果你習慣性地在每個提示詞裡都寫上 `send_message`，會讓任務的意圖更難讀懂。比較乾淨的做法是把投遞交給排程器處理，讓提示詞專注在任務本身。

**任務讀取的資料可能夾帶提示詞注入。** 如果你的 cron job 抓取一個網頁然後總結它，那個網頁理論上可以包含試圖重新導向 Agent 行為的指令。Hermes 在建立和更新時會掃描 cron 提示詞中的注入模式，但它不會過濾任務在執行時拉進來的外部內容。這屬於 OWASP 在 LLM Top 10 中描述的間接提示詞注入風險——對於任何會讀取不受信任輸入的 cron job，都值得放在心裡。對低風險的個人任務要怎麼看待這件事，我還沒有完全想清楚，但對於任何涉及憑證或外部系統的任務，這很重要。

## 值得使用的操作模式

以下是一些我一直在用的配置：

**固定工作時間的每日報告。** `0 9 * * 1-5`，搭配一個拉取資料的技能，以及一個準確說明要總結什麼的自包含提示詞。

**定期健康檢查。** `*/15 * * * *`——每 15 分鐘——執行一個小腳本 ping 某個端點，只有在出問題時才發送訊息。Cron job 中的 `script` 欄位在每次 Agent 執行前運行，它的 stdout 會變成 Agent 的上下文。這對於變更檢測模式非常有用，你不想要一天 96 次吵死人的輸出。

**多技能任務。** 附加兩三個技能，讓一個任務結合資料收集技能和分析技能。Cron 提示詞就變成「同時使用這兩個技能並報告」。

**有備援意識的任務。** Cron job 會繼承配置好的備援提供商和憑證池輪換機制。所以如果一個任務在凌晨 3 點執行時主提供商回傳了速率限制錯誤，它可以路由到備援提供商而不是讓整個任務失敗。這是我在一開始沒怎麼在意的功能，直到它救了我一個本來會錯過的執行。

我一直在告訴自己的事情是：**一個會跑的 cron job，比一個優雅的 cron job 更有價值。** 無聊、囉嗦、自包含的提示詞。明確的排程。以服務形式安裝的 Gateway。負責做重活的技能。這個組合才是讓「我設了這個」和「這個已經安安靜靜跑了三個星期我差點忘了它的存在」之間的差別。

## 常見問題

**為什麼我的 cron job 沒有之前的對話歷史？**

每次執行都是一個全新的 Agent 對話，這是設計。沒有歷史，沒有記憶繼承。提示詞和附加的技能是 Agent 看到的唯一上下文。

**我需要在提示詞裡呼叫 send_message 嗎？**

不需要，而且最好別這麼做。排程器會自動投遞 Agent 的最終回應。如果你的提示詞也呼叫了 `send_message` 到同一個目標，Hermes 會檢測到重複然後跳過第二次發送。

**我的任務已經排程了但從未觸發，哪裡出了問題？**

最常見的原因是 Gateway 沒有在運行。試試 `hermes cron status` 檢查排程器。Cron job 只有當 Gateway 啟動時才會觸發，或者在 CLI 模式下有活躍的對話正在運行。

**Cron job 可以創建更多的 cron job 嗎？**

不行。Cron 執行的對話中禁用了 cronjob 工具集，以防止失控的排程循環。

**Cron job 可以執行多久？**

預設的超時是基於非活躍狀態，而不是牆上時間——一個正在呼叫工具或串流 token 的任務可以無限執行。一個閒置時間超過 `HERMES_CRON_TIMEOUT`（預設 600 秒）的任務會被終止。

---

我大概會繼續改進我寫這些提示詞的方式。Cron 這種東西，大部分讓它正常運作的因素並不在排程欄位裡——它在提示詞裡、在技能裡、在底層系統是否真的在運行裡。值得時不時檢查一下，這三樣東西是不是都還在原來的位置。
