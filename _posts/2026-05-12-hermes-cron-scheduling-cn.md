嗨，我是 Lena。我的第一个 Hermes Cron Job 大概跑了三天，我才发现它根本没在做我以为它在做的事。

它在清单里看起来一切正常。排程是对的，提示词也是某个星期二下午写的，看着挺清楚。但执行的结果总感觉有点歪——不是说坏掉了，就是……跟我在平常对话里问同一个问题得到的回答不太一样。

我停下来想了想。我一直在把它当成 Linux 的 cron 来用。问题是，**Hermes Agent 的 cron 根本不是 Linux cron**。它只是借用了那个排程语法而已。

这篇文章记录了我慢慢搞懂 Hermes 里 cron 到底是怎么运作的过程——以及为什么那么多任务看起来排程正确，执行起来却不对劲。这种事情大概不急的时候都不觉得是问题，等觉得是问题的时候就来不及了。

---

## Hermes Cron 到底在做什么

第一件值得大声说清楚的事：**Hermes 的 cron job 不是一个计时器上的 shell 指令。它是一个按照排程触发的 Agent 会话。**

这个区别比我想象的重要得多。根据官方文档，每次排程执行都在一个**全新的 Agent 会话**里开始。没有任何对话历史。没有上一次执行的记忆。没有从你平常互动中带过来的上下文。Agent 执行工作需要的所有东西，都得靠提示词提供——或者靠附加的技能（Skills）来提供。

第二件事：**Gateway 必须在运行**。排程器不是一个可以忽略的独立守护进程。在 Gateway 模式下，排程器的 tick 是 Gateway 主事件循环的一部分，大概每 60 秒检查一次。如果 Gateway 没有启动，你的任务就静静躺在 `jobs.json` 里，看起来排程完美，但什么都不做。这个我是意外发现的——一个看起来"没有在跑"的任务其实没问题，只是 Gateway 两天前挂掉了。

第三件事是关于投递（Delivery）的。Agent 的最终回应会自动投递到配置好的目标。所以如果你在提示词结尾写了"把这个发到 Telegram"，而目标**确实**是 Telegram，Hermes 会检测到重复然后只发一次。这部分不用你操心——但你得知道它在做这件事。

## 排程格式、重复、技能与投递目标

Hermes 同时支持自然语言间隔和标准的 5 字段 POSIX cron 表达式。所以 `every 1h`、`30m` 以及 `0 9 * * 1-5` 都可以用。

5 字段格式跟其他地方用的是一样的——Linux crontab、Kubernetes CronJobs、GitHub Actions——而 Wikipedia 上关于 cron 的条目其实是个出乎意料好用的参考资料，如果你想确认 Vixie 方言（Hermes 也在用的那个）里步进值和范围到底是怎么互动的话。

如果你不确定自己的表达式到底是不是你想的那样，`crontab.guru` 是我一直回头用的工具。把表达式贴进去，读它的中文解说，检查接下来 5 次执行时间。这种事花 30 秒就能省下一个除错的下午。

你可以附加技能（`--skill name`，一个或多个），用 `--deliver` 设定投递目标，给任务取个名字，还可以按任务覆盖模型。任务记录以 JSON 格式储存在 `~/.hermes/cron/jobs.json` 里，使用原子写入——意思是如果写入被中断，你也不会拿到一个半坏的文件。

## 创建经得起实际使用的任务

我最后常用的模式大概是这样的：

```bash
hermes cron create "0 9 * * 1-5" \
  "从 ~/work/project 拉取昨天的提交，\
   用 5 个要点总结变更内容，并标出任何看起来异常的地方" \
  --skill git-summary \
  --name "工作日站会准备"
```

这里我想特别说三个点：

**1. 排程是明确的。** `0 9 * * 1-5`——早上 9 点，工作日。不是每天早上 9 点。这个 cron 表达式对任何懂语法的人来说都不会误解，包括半夜 11 点试图 debug 为什么星期六触发了的我自己。

**2. 提示词是自包含的。** 它说了 repo 在哪。它说了要总结什么。它说了要几个要点。它没有假设 Agent 记得之前的任何东西。

**3. 附加了技能。** `git-summary` 技能（这里是假设的）承载了具体的操作步骤——怎么读 git log、用什么格式、什么算"异常"。Cron 的提示词只需要说**做什么**，不用说**怎么做**。

第三点是我一直在反复学习的东西。

## 为什么自包含的提示词比你想像的重要

这里有一个我第一周写的提示词，它没有按照我预期的方式工作：

> "跟上次一样，用这周的数据。"

在我平常的互动对话里，这个提示词完全没问题。Agent 有对话历史，知道"上次"是什么，知道我们在看什么数据。但在 cron 执行中，这些都不存在。全新的会话。没有记忆。没有上下文。Agent 读到"跟上次一样"，完全没有东西可以比较。

一个真正能用的版本是这样的：

> "读取 ~/data/weekly-signups.csv。计算相较于前 7 天的注册人数变化百分比。报告数字、变化量以及前 3 大来源管道。格式化为 Slack 消息。"

无聊。啰嗦。具体。每次都管用。

官方文档里有一个很好的例子，推荐的模式本质上是：**假设 Agent 是第一次读你的提示词，在隔离环境中，除了附加的技能之外没有任何上下文。** 如果这个提示词对一个陌生人来说不能作为一个独立的指令，那么它对一个全新的 cron 会话也同样不合适。

## 常见的 Cron 失败模式与预防方法

以下是一些我看过出问题的模式，大部分是我自己踩的坑：

**Gateway 没在运行的沉默。** 任务已排程、清单有显示、状态看起来正确，但什么都没触发。解决方法通常是：用 `hermes cron status` 确认排程器还活着，然后确保 Gateway 以服务形式安装，这样重启或登出后还能继续跑。在你把它安装成用户或系统服务之前，每一个 `hermes gateway` 前台执行都离失败只差一个终端窗口关闭。

**带有隐含上下文的脆弱提示词。** 这个已经说过了。我最常看到的版本是"总结一下有什么新的"——跟什么比？什么时候？Agent 不知道。

**因频率误判导致任务重叠。** 一个设定为每 1 分钟执行一次但需要 90 秒才能完成的任务，会产生一个有趣的问题。Hermes 在排程器的 tick 上使用文件锁定，防止同一批到期的任务被并行处理两次，这很好。但更深层的问题是：你**真的**需要 1 分钟的精确度吗？通常不需要。

**提示词要求投递导致的重复发送。** 如果你的提示词结尾写了 `send_message(...)` 到排程器已经要投递的同一个目标，Hermes 会检测到重复然后跳过第二次发送。这没问题。但如果你习惯性地在每个提示词里都写上 `send_message`，会让任务的意图更难读懂。比较干净的做法是把投递给排程器处理，让提示词专注在任务本身。

**任务读取的数据可能夹带提示词注入。** 如果你的 cron job 抓取一个网页然后总结它，那个网页理论上可以包含试图重新导向 Agent 行为的指令。Hermes 在创建和更新时会扫描 cron 提示词中的注入模式，但它不会过滤任务在执行时拉进来的外部内容。这属于 OWASP 在 LLM Top 10 中描述的间接提示词注入风险——对于任何会读取不受信任输入的 cron job，都值得放在心里。对低风险的个人任务要怎么看这件事，我还没有完全想清楚，但对于任何涉及凭证或外部系统的任务，这很重要。

## 值得使用的操作模式

以下是一些我一直在用的配置：

**固定工作时间的每日报告。** `0 9 * * 1-5`，搭配一个拉取数据的技能，以及一个准确说明要总结什么的自包含提示词。

**定期健康检查。** `*/15 * * * *`——每 15 分钟——执行一个小脚本 ping 某个端点，只有在出问题时才发送消息。Cron job 中的 `script` 字段在每次 Agent 执行前运行，它的 stdout 会变成 Agent 的上下文。这对于变更检测模式非常有用，你不想要一天 96 次吵死人的输出。

**多技能任务。** 附加两三个技能，让一个任务结合数据收集技能和分析技能。Cron 提示词就变成"同时使用这两个技能并报告"。

**有备份意识的任务。** Cron job 会继承配置好的备份提供商和凭证池轮换机制。所以如果一个任务在凌晨 3 点执行时主提供商返回了速率限制错误，它可以路由到备份提供商而不是让整个任务失败。这是我一开始没怎么在意的功能，直到它救了我一个本来会错过的执行。

我一直在告诉自己的事情是：**一个会跑的 cron job，比一个优雅的 cron job 更有价值。** 无聊、啰嗦、自包含的提示词。明确的排程。以服务形式安装的 Gateway。负责做重活的技能。这个组合才是让"我设了这个"和"这个已经安安静静跑了三个星期我差点忘了它的存在"之间的差别。

## 常见问题

**为什么我的 cron job 没有之前的对话历史？**

每次执行都是一个全新的 Agent 会话，这是设计。没有历史，没有记忆继承。提示词和附加的技能是 Agent 看到的唯一上下文。

**我需要在提示词里调用 send_message 吗？**

不需要，而且最好别这么做。排程器会自动投递 Agent 的最终回应。如果你的提示词也调用了 `send_message` 到同一个目标，Hermes 会检测到重复然后跳过第二次发送。

**我的任务已经排程了但从未触发，哪里出了问题？**

最常见的原因是 Gateway 没有在运行。试试 `hermes cron status` 检查排程器。Cron job 只有当 Gateway 启动时才会触发，或者在 CLI 模式下有活跃的会话正在运行。

**Cron job 可以创建更多的 cron job 吗？**

不行。Cron 执行的会话中禁用了 cronjob 工具集，以防止失控的排程循环。

**Cron job 可以执行多久？**

预设的超时是基于非活跃状态，而不是墙上时间——一个正在调用工具或串流 token 的任务可以无限执行。一个闲置时间超过 `HERMES_CRON_TIMEOUT`（默认 600 秒）的任务会被终止。

---

我大概会继续改进我写这些提示词的方式。Cron 这种东西，大部分让它正常运作的因素并不在排程字段里——它在提示词里、在技能里、在底层系统是否真的在运行里。值得时不时检查一下，这三样东西是不是都还在原来的位置。
